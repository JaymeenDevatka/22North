1. Executive Summary & Problem Statement

In rapidly expanding software-as-a-service (SaaS) companies, cloud expenditure often grows exponentially, frequently outstripping revenue growth and squeezing gross margins. The platform 22North was conceived to address these core challenges:

  • Cost Waste: Typically, 30–35% of cloud infrastructure spend is wasted on over-provisioned (low-utilization) or entirely idle resources.
  • Lack of Visibility: Finance and engineering teams operate in silos, lacking a single dashboard showing service spend broken down by owner teams, environments, and regions.
  • Integration Friction: Connecting third-party optimization tools directly to live cloud accounts (AWS/Azure) introduces security, compliance, and overhead bottlenecks, delaying insights.

Key Question Answered: How can growing SaaS companies achieve immediate, actionable visibility and cost savings without the risk and complexity of live cloud read/write permissions?

2. The Solution Design: 22North

22North operates on a zero-friction ingestion model. Finance or engineering managers export standard billing usage metadata into a CSV and upload it to the platform. The platform runs these inputs through an intelligent rule-based cost-optimisation engine to deliver prioritized, high-confidence recommendations.

3. System Architecture

The system is designed with a lightweight React-based frontend, a Node.js REST API layer, an isolated JavaScript-based Insight Engine, and a MySQL persistence store (with a robust local fallback mechanism).

Architectural Component Details

  • Frontend (React/Vite): A highly responsive UI displaying financial cards, cost-breakdown bar charts, priority filters, and an upload wizard. Styling relies on custom CSS incorporating visual assets, smooth micro-interactions, and high readability.
  • REST API Layer (Express): Serves JSON endpoints for checking service health, querying parsed dashboards, tracking upload history, and conducting transactional CSV imports.
  • Insight Engine: An inline logic engine that runs deterministic heuristics over the parsed resources, calculating utilization-to-cost indexes, sorting savings, mapping efforts, and setting action confidence scores.
  • Database (MySQL / Seed Fallback): Persists accounts, resource details, and import logs. When MySQL is unavailable (e.g. local standalone run), the API seamlessly falls back to seed data to ensure continuous demonstration.

5. Key Assumptions & Design Trade-offs

Design Assumptions

  • Ingested Formats: The SaaS team has access to billing reports containing columns representing Service, Cost, and Utilization. The system provides flexible header aliasing to match typical formats.
  • MVP Scope: Live cloud account permissions (which require API secrets and security approvals) are bypassed. In-memory analytical processing is highly scalable for datasets under 100k rows.

Architectural Trade-offs

Decision Area Design Choice Trade-off Rationale
Analytics Rule-Based Heuristics Opaque ML engines are harder to inspect. Plain rule-based logic is immediately explainable, fast, and deterministic for cloud resources.
Ingestion In-Memory CSV Processing CSV data is loaded, analyzed, and returned without database bloat. MySQL is used solely to audit import runs and accounts.
Integrations Zero Live AWS APIs Avoids live credential management, API throttling, and security reviews, leading to an immediate 10-minute setup time.
